# TradingAgents-Astock 商业化落地方案

## 1. 目标与边界

目标是在保持核心项目开源的前提下，形成能够覆盖维护、发布、客服和基础设施成本的收入。商业价值来自可靠交付和持续服务，不来自承诺选股收益。

本方案基于以下前提：

- 核心仓库继续采用 Apache 2.0。
- 免费用户可以获取、修改和运行源码。
- 官方构建、自动更新、托管任务、报告模板、技术支持和私有部署可以收费。
- 产品定位为研究与决策辅助工具，不提供保收益、代客理财或个性化证券投资建议。

## 2. 当前实现审计

仓库已经具备试用、激活码、设备绑定、付费页面拦截和管理员发证的雏形，但还不能直接用于正式收费。

### 上线阻断项

1. 客户端包含 HMAC 签名密钥和激活码生成函数。代码开源后，任何人都可以生成有效激活码。
2. 存在默认管理员账号和固定密码摘要。正式发布时必须删除默认凭据，只允许通过环境变量或首次初始化设置管理员。
3. 用户库和试用记录位于本机 SQLite。删除本地目录即可重新试用，也无法跨设备统一管理订单与授权。
4. Premium 页面仅在本地 Streamlit 代码中拦截。开源用户可以自行修改，因此它只能作为官方构建的便利功能，不能作为强 DRM。
5. 当前页面展示的微信、Telegram、价格和“永久买断”尚未形成经过验证的订单、退款和服务流程。
6. `PROMO.md` 仍有编码损坏和占位信息，不能直接用于公开宣传。

### 已验证基础

- 授权输入已有基础规范化和参数化 SQLite 查询。
- 设备数量、到期月份和试用截止日期已有测试覆盖。
- 2026-06-18 完整测试基线为 152 项通过。

## 3. 推荐收入结构

### 社区版：免费

- GitHub 源码与社区安装方式。
- 本地运行、自备模型 API Key。
- 基础数据连接、核心 Agent 框架和安全更新。
- GitHub Issue 社区支持，不承诺响应时效。

### 支持者版：99 元/年

- 官方 Windows/macOS 构建包。
- 国内下载镜像和稳定更新通道。
- 官方配置向导、报告模板和提示词模板。
- 支持者群和版本提前体验。

该层权益应尽量选择边际成本低的内容。可以提供限量早期支持者永久权益，但不得包含永久云资源或无限人工支持。

### Pro 版：39 元/月或 299 元/年

- 定时生成自选股日报、周报。
- 报告历史、组合观察列表和通知任务。
- 高级研究模板与批量工作流。
- 自备模型 Key，避免价格被模型调用成本吞噬。

不建议继续使用“99 元/月、299 元永久”的正式定价。永久价格只有月费的三倍，会迅速消耗未来收入和支持能力。

### 托管版：99 元/月起，模型用量另计

- 无需本地部署的托管任务。
- 云端调度、报告存储、邮件或企业微信通知。
- 明确计算、存储和模型 Token 配额。
- 不提供永久买断。

### 私有部署：2999 元起

- 企业或团队私有部署。
- 指定模型、数据源和内部通知系统接入。
- 部署培训、升级服务和可选 SLA。
- 定制开发单独评估，不包含在低价会员中。

## 4. 权益设计原则

| 能力 | 社区版 | 支持者版 | Pro/托管版 |
|---|:---:|:---:|:---:|
| 源码与本地 Agent 框架 | 是 | 是 | 是 |
| 自备 Key 本地分析 | 是 | 是 | 是 |
| 官方安装包与稳定更新 | 否 | 是 | 是 |
| 官方高级模板 | 否 | 是 | 是 |
| 定时任务与报告历史 | 本地自行配置 | 本地 | 托管 |
| 人工支持 | 社区 | 群内支持 | 优先支持 |
| 私有部署 | 否 | 否 | 单独报价 |

开源许可证授予的权利不能用“赞助”名义收回。收费权益应围绕官方交付、服务质量和运营成本设计。

## 5. 推荐授权架构

### 第一阶段：赞助信任制

- 保留轻量本地提示，不把它描述为不可绕过的 DRM。
- 通过 GitHub Sponsors、爱发电或合规订单平台记录支持者。
- 付费交付官方构建、更新通道和模板包。

这是最接近 go-stock 的方式，开发成本最低，也更符合开源社区预期。

### 第二阶段：离线签名授权

- 管理端离线保存 Ed25519 私钥，不进入仓库和安装包。
- 客户端只包含公钥，用于验证包含用户、套餐、到期日和设备数的签名许可证。
- 删除客户端的通用发证函数和硬编码共享密钥。
- 私钥轮换、吊销和设备重置通过独立管理工具完成。

离线签名能防止用户直接伪造许可证，但无法阻止修改开源客户端。因此它适合官方安装包的授权体验，不应被描述为绝对防破解。

### 第三阶段：托管权益服务

- 订单平台支付成功后由服务端发放权益。
- 服务端统一管理订阅、试用、设备、吊销和退款状态。
- 客户端只保存短期令牌或签名许可证，不保存发证秘密。
- 登录、验证码和激活接口必须限流，并记录不含密钥的安全审计日志。

## 6. 合规与产品文案

- 将“AI 荐股”逐步改为“AI 研究候选”或“AI 信号研究”。
- 不使用“稳赚、胜率保证、今日必涨、带单”等表述。
- 报告必须显示数据时间、来源、模型局限和风险提示。
- 收费前准备服务条款、隐私政策、退款政策和数据源使用条款核对。
- 对外开展付费证券分析或个性化建议前，应取得专业法律意见并确认所需资质。

## 7. 四周落地顺序

### 第 1 周：可信发布

- 修复 README、宣传文案和真实下载地址。
- 公布免费与付费权益边界。
- 确认唯一官方收款与客服渠道。
- 删除默认管理员凭据。

验收：新用户能够从 README 完成安装，并能识别唯一官方渠道。

### 第 2 周：安全签发

- 用非对称签名替换公开 HMAC 密钥。
- 管理端发证工具与公开客户端分离。
- 增加伪造、过期、错误用户和错误设备测试。

验收：只有持有私钥的管理端能够生成客户端认可的许可证。

### 第 3 周：支持者交付

- 自动构建 Windows 安装包。
- 建立版本清单、校验值和稳定更新通道。
- 制作第一批官方模板和安装排障文档。

验收：支持者可在 15 分钟内完成安装与首次报告。

### 第 4 周：验证付费意愿

- 小范围招募 20 名早期用户。
- 记录激活率、次周留存、报告运行成功率和客服耗时。
- 根据使用数据决定是否开发托管服务，不预先建设重型后端。

验收：至少有真实付费转化，并能量化每名付费用户的支持成本。

## 8. 首批指标

- README 到安装成功转化率。
- 试用激活率和 7 日核心功能使用率。
- 试用到年付转化率。
- 每周成功生成报告的用户比例。
- 单个付费用户月均客服时长。
- 退款率、任务失败率和续费率。

先验证用户是否愿意为“稳定安装、自动报告和持续支持”付费，再建设支付后端和云端集群。
